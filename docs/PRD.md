# 产品需求文档

## 1. 产品概述

- 产品名称：Atoms Demo（代号可自定，如 Isotope）
- 一句话描述：一个可公网体验的多智能体应用生成 Demo，支持对话驱动生成、模式切换、任务协作与实时预览。
- 产品目标：
  - 在有限时间内交付可运行、可体验、可扩展的 Atoms 风格产品原型。
  - 验证「智能体驱动生成 + 可视化预览 + Team 协作编排」的核心闭环。
  - 满足 ROOT 全栈笔试对真实交互、数据持久化与在线可访问的要求。
- 目标用户：
  - 笔试评审人（主要）：通过内置账号登录，完整体验主流程。
  - 内部演示观众：快速理解 Atoms 类产品的交互形态。

## 2. 用户场景与核心需求

- 用户画像：
  - 评审人：有限时间体验 Demo，关注产品完整性、工程闭环与交互可信度。
  - 演示者：需要稳定账号、可重复演示路径，避免公网开放注册带来的滥用风险。
- 使用场景：
  1. 登录内置账号，进入首页，选择 Engineer / Team 模式后发起生成。
  2. 在项目页与智能体对话，持续迭代应用；右侧 App Viewer 实时预览。
  3. Team 模式下由 Leader（Mike）创建并分配任务给工程师（Alex），改码后自动构建并刷新预览。
  4. 刷新浏览器后，项目、对话与版本记录仍可恢复。
- 核心痛点：
  - 传统 Demo 多为静态页或纯 PoC，难以体现多智能体协作与构建预览闭环。
  - 公网开放注册会引入垃圾账号与滥用风险，不适合笔试 Demo。
- 核心需求：
  - 智能体驱动生成应用，并以可视化网页展示。
  - 真实交互 + 数据持久化 + 可测试在线链接。
  - Engineer / Team 模式可切换，Team 具备任务分配。
  - 代码变更后自动构建，App Viewer 实时更新。
  - 内置账号登录，不开放注册。

## 3. 功能范围

### P0（必须实现）

- 内置账号登录（无公开注册）
- 首页与项目入口
- 对话驱动应用生成（多轮）
- App Viewer 实时预览
- 项目 / 会话 / 生成结果持久化
- 模式切换：Engineer / Team
- Team 模式：Leader 任务分配与完成
- 自动构建与预览实时更新
- 轻量版本记录（版本卡片）

### P1（重要功能，时间允许优先）

- 工作区视图壳：编辑器（只读文件树 + 代码查看）
- 工作区视图壳：文件列表（只读）
- 预览区「设计 / 控制台」子页
- 预览响应式视口切换（桌面 / 平板 / 手机）
- 发布 / 分享入口占位（禁用或说明 Demo 未开放）

### P2（后续迭代）

- Race（竞赛）模式
- Deep Research
- 文件 / 文件夹上传与 `#` 引用
- Issue Report 一键修复
- 真实 Publish（自定义域名 / SEO / Ads）
- Remix / 完整版本分支
- 开放注册与权限体系
- Atoms Cloud（真实 DB / Auth / Stripe）
- SEO 智能体 Sarah、广告智能体等增长能力

## 4. 页面与交互需求

- 页面列表：
  1. 登录页
  2. 首页（创建入口 + 我的项目）
  3. 项目工作台（聊天 + 预览 / 可选编辑器、文件）
- 页面目标：
  - 登录页：完成鉴权，阻挡未授权访问。
  - 首页：快速发起生成，进入已有项目。
  - 项目工作台：完成「对话 → 分配（Team）→ 改码 → 构建 → 预览」闭环。
- 关键流程：

```text
访问 Demo URL
  → 未登录：登录页（内置账号）
  → 已登录：首页
      → 选择模式 + 输入需求 → 创建项目并进入工作台
      → 或从「我的项目」进入已有项目
  → 工作台：
      Engineer：直接与工程师对话改码 → 自动构建 → App Viewer 更新
      Team：Mike 创建任务 → @Alex 执行 → 完成任务 → 自动构建 → App Viewer 更新
  → 登出后回到登录页，核心能力不可用
```

- 交互要求：
  - 生成 / 构建中需有明确加载与进度反馈，禁止无响应空白。
  - 构建成功 / 失败需在聊天流或预览区给出可读提示。
  - 当前模式（Engineer / Team）在输入区或顶栏始终可见。
  - Team 模式任务卡片需可点击「创建任务」「完成任务」（Demo 可模拟确认流）。
  - 移动端至少保证登录、对话、预览可读可用（布局可简化）。

## 5. 产品风格与体验要求

- 整体风格：
  - 对齐 Atoms 的「左侧对话 + 右侧工作区」产品形态，清晰、克制、偏工具感。
  - 不追求像素级复刻官方品牌资产；重点还原信息架构与主流程体验。
- 视觉要求：
  - 浅色主界面，主操作按钮有明确强调色。
  - Agent 消息需展示角色名与身份标签（如 Mike | 团队领导、Alex | 工程师）。
  - 版本卡片、任务卡片与普通文本消息有区分。
- 交互体验：
  - 关键操作（发送、切模式、完成任务）有即时反馈。
  - 预览刷新平滑；构建失败可重试或通过继续对话修复。
  - 避免堆砌无关营销入口（积分、社区、升级可隐藏或禁用）。
- 适配要求：
  - 桌面端（≥1280px）为完整双栏体验。
  - 平板 / 窄屏可改为上下堆叠或抽屉式聊天。
  - 公网 HTTPS 可访问；基础浏览器（Chrome / Safari 近两年版本）可用。

## 6. 业务规则

### 6.1 账号与安全

- 规则 A：不提供注册入口；仅允许配置的内置账号登录。
- 规则 B：未登录用户不能新建项目、发送对话、触发生成 / 构建。
- 规则 C：内置账号与密码通过环境变量或部署配置注入，可在 README 中提供演示账号说明。
- 规则 D：支持登出；登出后清除会话凭证。

### 6.2 模式切换

- 规则 E：支持 Engineer 与 Team 两种模式；新建项目时可选择，项目内可切换。
- 规则 F：
  - Engineer：主要由工程师 Agent 直接执行修改，不强制走任务分配。
  - Team：必须体现 Leader（Mike）拆解 / 创建任务，并指派给执行者（至少 Alex）。
- 规则 G：当前模式需持久化到项目（例如记录 last mode），刷新后保持。

### 6.3 生成、构建与预览

- 规则 H：用户自然语言输入后，系统产生可预览的前端应用结果（真实生成或高质量受控演示均可，但必须可交互，非纯静态截图）。
- 规则 I：Agent 完成代码修改后，系统自动触发构建 / 构建检查。
- 规则 J：构建成功 → App Viewer 自动刷新为最新结果；构建失败 → 明确错误提示，不静默失败。
- 规则 K：每次成功构建可生成轻量「版本卡片」（版本号 + 一句话变更摘要）。

### 6.4 数据持久化

- 规则 L：至少持久化：用户会话、项目列表、对话消息、模式、版本摘要、当前预览关联状态。
- 规则 M：技术方案不限（SQLite / Postgres / 本地文件库等），但刷新与重新登录后数据可恢复。
- 规则 N：公网多用户共用 Demo 时，数据按登录用户隔离（至少项目归属到当前用户）。

### 6.5 范围边界

- 规则 O：本阶段不做开放注册、Publish 上线配置、Race、Deep Research、上传、Remix 完整能力。
- 规则 P：若实现编辑器 / 文件视图，默认只读，不要求在线改码保存。

## 7. 功能点明细与验收标准

> 以下为每个功能点的需求说明与验收标准（AC）。全部 P0 功能点必须通过。

### 7.1 内置账号登录

- 需求说明：提供登录页；仅内置账号可登录；无注册。
- 验收标准：
  - AC1：访问受保护路由时，未登录会跳转登录页。
  - AC2：正确账号密码可登录并进入首页。
  - AC3：错误密码给出明确失败提示，不进入系统。
  - AC4：页面无可达的「注册」入口。
  - AC5：登出后无法继续调用生成 / 构建接口（前端拦截 + 后端鉴权）。

### 7.2 首页与项目入口

- 需求说明：登录后展示创建入口与项目列表。
- 验收标准：
  - AC1：可见输入框与模式选择（Engineer / Team）。
  - AC2：提交需求后创建项目并进入工作台。
  - AC3：已有项目列表可点击进入。
  - AC4：空状态有引导文案（尚无项目时）。

### 7.3 对话驱动应用生成

- 需求说明：通过聊天让 Agent 生成 / 修改应用。
- 验收标准：
  - AC1：发送首条需求后，聊天区出现 Agent 回复，并产生可预览结果。
  - AC2：支持至少一轮「继续修改」对话，预览内容随之变化。
  - AC3：发送中输入框有禁用或 loading，避免重复提交混乱。
  - AC4：Agent 消息展示角色身份（至少工程师；Team 下还有团队领导）。

### 7.4 App Viewer 实时预览

- 需求说明：右侧（或主工作区）展示生成应用的可视化预览。
- 验收标准：
  - AC1：构建成功后可在 App Viewer 看到可交互页面（非纯图片占位）。
  - AC2：支持手动刷新预览。
  - AC3：构建 / 加载中有明确状态；超过合理时间有失败或重试提示。
  - AC4（P1）：支持桌面 / 移动视口切换则加分，非 P0 阻断项。

### 7.5 数据持久化

- 需求说明：项目与对话等关键数据可持久保存。
- 验收标准：
  - AC1：创建项目并产生对话后，刷新页面，项目与历史消息仍在。
  - AC2：登出再登录同一账号，仍可看到该用户项目。
  - AC3：预览可恢复到最近一次成功构建对应的状态（或可再次打开最近版本）。

### 7.6 模式切换（Engineer / Team）

- 需求说明：两种模式可切换，行为可区分。
- 验收标准：
  - AC1：首页与工作台均可看到当前模式，并可切换。
  - AC2：切换为 Engineer 后，主流程为工程师直接执行，不强制出现「创建任务」卡片。
  - AC3：切换为 Team 后，主流程出现 Leader 工作流与任务分配相关 UI。
  - AC4：刷新后当前模式保持不变。

### 7.7 Team 模式：Leader 任务分配

- 需求说明：Mike 作为团队领导创建任务并指派给 Alex 等执行者。
- 验收标准：
  - AC1：Team 模式下，用户提出修改需求后，出现 Mike 的工作流程说明。
  - AC2：可出现「创建任务」操作，并生成指向执行者的任务（含 `@Alex` 或等价指派信息）。
  - AC3：执行者完成后，Mike 侧可「完成任务」，任务状态变为已完成。
  - AC4：聊天流中能看到「已处理 N 步」或等价进度信息（文案可简化）。
  - AC5：同一需求在 Engineer 与 Team 下的交互路径肉眼可区分。

### 7.8 自动构建与预览实时更新

- 需求说明：代码修改后自动构建，成功则刷新 App Viewer。
- 验收标准：
  - AC1：Agent 声明完成代码修改后，无需用户手动点「构建」，系统自动进入构建。
  - AC2：构建成功时，聊天或系统消息提示可在 App Viewer 预览。
  - AC3：App Viewer 在构建成功后自动更新为新内容（允许短延迟，但需自动发生）。
  - AC4：构建失败时展示失败原因或日志摘要，预览不假装成功。

### 7.9 轻量版本记录

- 需求说明：成功变更后展示版本卡片。
- 验收标准：
  - AC1：至少一次成功构建后，聊天流出现「版本 N」卡片。
  - AC2：卡片包含简短变更摘要（可由 Agent 生成）。
  - AC3：版本信息写入持久化存储，刷新后仍可见。

### 7.10 工作台主界面骨架

- 需求说明：还原 Atoms 风格的双栏工作台。
- 验收标准：
  - AC1：左侧为对话区，右侧为工作区（默认 App Viewer）。
  - AC2：顶栏可见项目名称；可返回首页或项目列表。
  - AC3：输入区固定在对话区底部，支持发送。
  - AC4（P1）：顶栏可切换「应用查看器 / 编辑器 / 文件」等视图；未实现的视图需有明确空态，不得静默空白报错。

## 8. 成功标准

- 标准 A（笔试硬性）：公网可访问；真实交互；数据持久化；智能体驱动生成 + 可视化展示；非纯静态 PoC。
- 标准 B（范围完成度）：第 7 节全部 P0 验收标准通过。
- 标准 C（模式可感知）：评审人 2 分钟内能区分 Engineer 与 Team 的差异。
- 标准 D（闭环完整）：任意一次成功修改，均可观察到「改码 → 自动构建 → 预览更新」。
- 标准 E（公网安全底线）：无开放注册；未登录不可用核心能力；演示账号可文档说明。
- 标准 F（演示稳定性）：按「登录 → 新建 → 对话生成 →（Team）任务分配 → 预览更新 → 刷新仍在」路径可重复演示成功。

## 9. 后续迭代方向

- 方向 A：补齐工作区只读编辑器 / 文件视图与发布入口占位，提升 UI 完整度。
- 方向 B：引入 Race 模式与 Deep Research，增强「多方案对比 / 调研驱动」能力。
- 方向 C：支持文件上传上下文、Issue 一键修复，提高迭代效率。
- 方向 D：真实 Publish、域名、SEO、权限与多人协作。
- 方向 E：接入更完整的多智能体角色（PM / 架构师 / 数据 / SEO）与可观测任务看板。

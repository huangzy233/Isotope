# 产品需求文档

## 1. 产品概述

- 产品名称：Atoms Demo（代号可自定，如 Isotope）
- 一句话描述：一个可公网体验的多智能体应用生成 Demo，支持对话驱动生成、模式切换（含 Plan 需求澄清）、任务协作、记忆系统与实时预览。
- 产品目标：
  - 在有限时间内交付可运行、可体验、可扩展的 Atoms 风格产品原型。
  - 验证「智能体驱动生成 + 可视化预览 + Team 协作编排 + Plan 需求澄清」的核心闭环。
  - 满足 ROOT 全栈笔试对真实交互、数据持久化与在线可访问的要求。
- 目标用户：
  - 笔试评审人（主要）：通过内置账号登录，完整体验主流程。
  - 内部演示观众：快速理解 Atoms 类产品的交互形态。

## 2. 用户场景与核心需求

- 用户画像：
  - 评审人：有限时间体验 Demo，关注产品完整性、工程闭环与交互可信度。
  - 演示者：需要稳定账号、可重复演示路径，避免公网开放注册带来的滥用风险。
- 使用场景：
  1. 登录内置账号，进入首页，按需打开 Plan / Team 开关（都关即为 Engineer）后发起生成。
  2. 在项目页与智能体对话，持续迭代应用；右侧 App Viewer 实时预览。
  3. Team 开启时由 Leader（Mike）创建并分配任务给工程师（Alex），改码后自动构建并刷新预览。
  4. Plan 开启时由 Pat（产品）提问澄清、给出推荐路径与需求规格说明；用户确认后关闭 Plan 并执行（仅 Plan → Alex；Plan∧Team → Mike→Alex）。
  5. 刷新浏览器后，项目、对话与版本记录仍可恢复。
- 核心痛点：
  - 传统 Demo 多为静态页或纯 PoC，难以体现多智能体协作与构建预览闭环。
  - 公网开放注册会引入垃圾账号与滥用风险，不适合笔试 Demo。
- 核心需求：
  - 智能体驱动生成应用，并以可视化网页展示。
  - 真实交互 + 数据持久化 + 可测试在线链接。
  - Plan / Team 开关可独立开关并组合（都关 = Engineer）；Team 具备任务分配；Plan 具备需求澄清确认后再执行。
  - Agent 长期记忆系统（产品级 + 用户级；细则待定）。
  - 代码变更后自动构建，App Viewer 实时更新。
  - 内置账号登录，不开放注册。

## 3. 功能范围

### P0（必须实现）

- 内置账号登录（无公开注册）
- 首页与项目入口
- 对话驱动应用生成（多轮）
- App Viewer 实时预览
- 项目 / 会话 / 生成结果持久化
- 模式开关：Plan / Team（可组合；都关 = Engineer）
- Team：Leader 任务分配与完成
- Plan：Pat 澄清需求 → 推荐路径 → 规格说明 → 用户确认后执行（可与 Team 组合则交 Leader）
- 记忆系统（Agent 长期记忆；产品级 / 用户级分类与细则稍后讨论）
- 自动构建与预览实时更新
- 轻量版本记录（版本卡片）
- 断线重连（进行中回合 / 任务状态在刷新或重新登录后可恢复，不因客户端断开误标「生成失败」）

### P1（重要功能，时间允许优先）

- 工作区视图壳：编辑器（只读文件树 + 代码查看）
- 工作区视图壳：文件列表（只读）
- 预览区「设计 / 控制台」子页
- 预览响应式视口切换（桌面 / 平板 / 手机）
- 发布 / 分享入口占位（禁用或说明 Demo 未开放）
- Agent 安全与权限治理（工具白名单 / 路径与命令边界 / 越权拒绝可观测）
- Prompt 支持指定模型（按 prompt meta / 配置声明模型，回合按声明路由）

### P2（后续迭代）

- Race（竞赛）模式
- Deep Research
- 文件 / 文件夹上传与 `#` 引用
- Issue Report 一键修复
- 真实 Publish（自定义域名 / SEO / Ads）
- Remix / 完整版本分支
- **版本 git 快照与回滚**（对 `workspace/` 打增量快照写入 `snapshotRef`；版本卡「恢复到此版本」；保留上限控制体积；见 `docs/superpowers/specs/2026-07-18-lightweight-version-cards-design.md` §10）
- 开放注册与权限体系
- Atoms Cloud（真实 DB / Auth / Stripe）
- SEO 智能体 Sarah、广告智能体等增长能力

## 4. 页面与交互需求

- 页面列表：
  1. 登录页
  2. 首页（创建入口 + 我的项目）
  3. 项目工作台（聊天 + 预览 / 可选编辑器、文件）
- 页面目标：
  - 登录页：完成鉴权，阻挡未授权访问。
  - 首页：快速发起生成，进入已有项目。
  - 项目工作台：完成「对话 →（Plan 澄清 / Team 分配）→ 改码 → 构建 → 预览」闭环。
- 主流程：

```text
访问 Demo URL
  → 未登录：登录页（内置账号）
  → 已登录：首页
      → 选择模式 + 输入需求 → 创建项目并进入工作台
      → 或从「我的项目」进入已有项目
  → 工作台：
      Engineer（Plan/Team 都关）：直接与工程师对话改码 → 自动构建 → App Viewer 更新
      Team：Mike 创建任务 → @Alex 执行 → 完成任务 → 自动构建 → App Viewer 更新
      Plan：Pat 提问澄清 → 推荐路径（1/2/3/4 或自定义）→ 需求规格说明
            → 用户 OK/确认 → 关闭 Plan → 静默执行（仅 Plan→Alex；Plan∧Team→Mike→Alex）
            → 自动构建 → App Viewer 更新
  → 登出后回到登录页，核心能力不可用
```

- 交互要求：
  - 生成 / 构建中需有明确加载与进度反馈，禁止无响应空白。
  - 构建成功 / 失败需在聊天流或预览区给出可读提示。
  - 当前 Plan / Team 开关状态在输入区可见（开启时以 Composer chip 展示，可点 X 关闭）；都关即为 Engineer。
  - Team 模式任务卡片需可点击「创建任务」「完成任务」（Demo 可模拟确认流）。
  - Plan 开启时 Pat 的提问、推荐路径与规格说明需可读；确认前不进入工程师改码。
  - 移动端至少保证登录、对话、预览可读可用（布局可简化）。

## 5. 产品风格与体验要求

- 整体风格：
  - 对齐 Atoms 的「左侧对话 + 右侧工作区」产品形态，清晰、克制、偏工具感。
  - 不追求像素级复刻官方品牌资产；重点还原信息架构与主流程体验。
- 视觉要求：
  - 浅色主界面，主操作按钮有明确强调色。
  - Agent 消息需展示角色名与身份标签（如 Mike | 团队领导、Alex | 工程师、Pat | 产品）。
  - 版本卡片、任务卡片与普通文本消息有区分。
- 交互体验：
  - 关键操作（发送、切模式、完成任务、确认 Plan 路径）有即时反馈。
  - 预览刷新平滑；构建失败可重试或通过继续对话修复。
  - 避免堆砌无关营销入口（积分、社区、升级可隐藏或禁用）。
- 适配要求：
  - 桌面端（≥1280px）为完整双栏体验。
  - 平板 / 窄屏可改为上下堆叠或抽屉式聊天。
  - 公网 HTTPS 可访问；基础浏览器（Chrome / Safari 近两年版本）可用。

## 6. 业务规则

### 6.1 账号与安全

- 规则 A：不提供注册入口；仅允许配置的内置账号登录。
- 规则 B：未登录用户不能新建项目、发送对话、触发生成 / 构建。
- 规则 C：内置账号与密码通过环境变量或部署配置注入，可在 README 中提供演示账号说明。
- 规则 D：支持登出；登出后清除会话凭证。

### 6.2 模式切换

- 规则 E：支持 Plan、Team 两个独立开关（可组合）；都关闭时为 Engineer。新建项目时可选择，项目内可切换。
- 规则 F：
  - Engineer（Plan/Team 都关）：主要由工程师 Agent 直接执行修改，不强制走任务分配，也不强制先走需求澄清。
  - Team：必须体现 Leader（Mike）拆解 / 创建任务，并指派给执行者（至少 Alex）。
  - Plan：必须先由 Pat（产品）根据用户初始输入通过提问收集需求，给出推荐路径（用户可选编号如 1/2/3/4，或自行定义），并输出需求规格说明供审阅；用户以 OK/确认等肯定性回复确认后，关闭 Plan 并开始执行——仅 Plan 时交给工程师（Alex）；Plan 与 Team 同时开启时交给 Leader（Mike）再走任务分配。
- 规则 G：当前 Plan / Team 开关状态需持久化到项目，刷新后保持。

### 6.2.1 记忆系统（占位）

- 规则 G1：系统需具备 Agent 长期记忆能力，至少区分产品级与用户级（细则、类型划分与读写时机稍后讨论并回写本 PRD）。
- 规则 G2：记忆按登录用户隔离；产品级记忆默认绑定项目（除非后续设计明确支持跨项目）。

### 6.3 生成、构建与预览

- 规则 H：用户自然语言输入后，系统产生可预览的前端应用结果（真实生成或高质量受控演示均可，但必须可交互，非纯静态截图）。
- 规则 I：Agent 完成代码修改后，系统自动触发构建 / 构建检查。
- 规则 J：构建成功 → App Viewer 自动刷新为最新结果；构建失败 → 明确错误提示，不静默失败。
- 规则 K：每次成功构建可生成轻量「版本卡片」（版本号 + 一句话变更摘要）。

### 6.4 数据持久化

- 规则 L：至少持久化：用户会话、项目列表、对话消息、模式、版本摘要、当前预览关联状态。
- 规则 M：技术方案不限（SQLite / Postgres / 本地文件库等），但刷新与重新登录后数据可恢复。
- 规则 N：公网多用户共用 Demo 时，数据按登录用户隔离（至少项目归属到当前用户）。

### 6.5 范围边界

- 规则 O：本阶段不做开放注册、Publish 上线配置、Race、Deep Research、上传、Remix 完整能力。
- 规则 P：若实现编辑器 / 文件视图，默认只读，不要求在线改码保存。
- 规则 Q：记忆系统的具体记忆类型、存储形态与注入策略以后续讨论结论为准；未定稿前不阻塞 Plan / Engineer / Team 主流程交付，但 P0 必须保留可落地的端口与最小可用行为（或明确阶段性 AC）。
- 规则 R（P1）：Agent 工具 / 路径 / 命令须受权限策略约束；越权拒绝须可观测。本条不等于 P2 的账号 RBAC / 开放注册。
- 规则 S（P1）：Prompt 可通过 meta / 配置指定模型；未指定则使用全局默认；模型绑定不硬编码进业务 TS Prompt 字符串。

## 7. 功能点明细与验收标准

> 以下为每个功能点的需求说明与验收标准（AC）。全部 P0 功能点必须通过；P1 功能点在时间允许实现时按对应 AC 验收。

### 7.1 内置账号登录

- 需求说明：提供登录页；仅内置账号可登录；无注册。
- 验收标准：
  - AC1：访问受保护路由时，未登录会跳转登录页。
  - AC2：正确账号密码可登录并进入首页。
  - AC3：错误密码给出明确失败提示，不进入系统。
  - AC4：页面无可达的「注册」入口。
  - AC5：登出后无法继续调用生成 / 构建接口（前端拦截 + 后端鉴权）。

### 7.2 首页与项目入口

- 需求说明：登录后展示创建入口与项目列表。
- 验收标准：
  - AC1：可见输入框与模式选择（Plan / Team 开关；都关为 Engineer）。
  - AC2：提交需求后创建项目并进入工作台。
  - AC3：已有项目列表可点击进入。
  - AC4：空状态有引导文案（尚无项目时）。

### 7.3 对话驱动应用生成

- 需求说明：通过聊天让 Agent 生成 / 修改应用。
- 验收标准：
  - AC1：发送首条需求后，聊天区出现 Agent 回复，并产生可预览结果。
  - AC2：支持至少一轮「继续修改」对话，预览内容随之变化。
  - AC3：发送中输入框有禁用或 loading，避免重复提交混乱。
  - AC4：Agent 消息展示角色身份（至少工程师；Team 下还有团队领导）。

### 7.4 App Viewer 实时预览

- 需求说明：右侧（或主工作区）展示生成应用的可视化预览。
- 验收标准：
  - AC1：构建成功后可在 App Viewer 看到可交互页面（非纯图片占位）。
  - AC2：支持手动刷新预览。
  - AC3：构建 / 加载中有明确状态；超过合理时间有失败或重试提示。
  - AC4（P1）：支持桌面 / 移动视口切换则加分，非 P0 阻断项。

### 7.5 数据持久化

- 需求说明：项目与对话等关键数据可持久保存。
- 验收标准：
  - AC1：创建项目并产生对话后，刷新页面，项目与历史消息仍在。
  - AC2：登出再登录同一账号，仍可看到该用户项目。
  - AC3：预览可恢复到最近一次成功构建对应的状态（或可再次打开最近版本）。

### 7.6 模式切换（Plan / Team 可组合）

- 需求说明：Plan 与 Team 为独立开关，可组合；都关为 Engineer。行为可区分。
- 验收标准：
  - AC1：首页与工作台均可看到并切换 Plan / Team；开启时 Composer 输入区有对应 chip，点 X 可关闭。
  - AC2：Plan/Team 都关（Engineer）时，主流程为工程师直接执行，不强制出现「创建任务」卡片，也不强制先走产品澄清。
  - AC3：仅开 Team 时，主流程出现 Leader 工作流与任务分配相关 UI。
  - AC4：打开 Plan 后，主流程先由 Pat 澄清需求，确认前不直接进入工程师改码；与 Team 同时开时，确认后走 Leader 任务流。
  - AC5：刷新后 Plan / Team 开关状态保持不变。

### 7.7 Team 模式：Leader 任务分配

- 需求说明：Mike 作为团队领导创建任务并指派给 Alex 等执行者。
- 验收标准：
  - AC1：Team 模式下，用户提出修改需求后，出现 Mike 的工作流程说明。
  - AC2：可出现「创建任务」操作，并生成指向执行者的任务（含 `@Alex` 或等价指派信息）。
  - AC3：执行者完成后，Mike 侧可「完成任务」，任务状态变为已完成。
  - AC4：聊天流中能看到「已处理 N 步」或等价进度信息（文案可简化）。
  - AC5：同一需求在 Engineer 与 Team 下的交互路径肉眼可区分。

### 7.8 自动构建与预览实时更新

- 需求说明：代码修改后自动构建，成功则刷新 App Viewer。
- 验收标准：
  - AC1：Agent 声明完成代码修改后，无需用户手动点「构建」，系统自动进入构建。
  - AC2：构建成功时，聊天或系统消息提示可在 App Viewer 预览。
  - AC3：App Viewer 在构建成功后自动更新为新内容（允许短延迟，但需自动发生）。
  - AC4：构建失败时展示失败原因或日志摘要，预览不假装成功。

### 7.9 轻量版本记录

- 需求说明：成功变更后展示版本卡片。
- 验收标准：
  - AC1：至少一次成功构建后，聊天流出现「版本 N」卡片。
  - AC2：卡片包含简短变更摘要（可由 Agent 生成）。
  - AC3：版本信息写入持久化存储，刷新后仍可见。

### 7.10 断线重连（进行中状态恢复）

- 需求说明：Agent / Team 回合或任务进行中时，用户刷新页面、关闭标签后重新进入、或登出再登录同一账号，进行中状态不丢失；不得因客户端断开 SSE 而把回合误记为「生成失败」。
- 验收标准：
  - AC1：回合进行中刷新工作台，仍能看到进行中指示（如 Streaming / Running / 占位或已落库的过程步骤），而不是空白或立刻失败。
  - AC2：客户端断开（刷新 / 关页）后，服务端回合若仍在跑，不因 SSE `Controller is already closed` 等传输错误把助手消息写成「生成失败：…」。
  - AC3：回合在断开期间正常结束后，再次进入工作台可看到最终助手结论（及过程信息）；若仍在进行，可恢复订阅或轮询看到后续进度直至结束。
  - AC4：Team 模式下进行中的任务状态（pending / assigned / running 等）刷新后仍正确；与回合锁/watchdog 行为一致、不出现「假死锁」或重复开跑（设计时钉死一种策略）。

### 7.11 工作台主界面骨架

- 需求说明：还原 Atoms 风格的双栏工作台。
- 验收标准：
  - AC1：左侧为对话区，右侧为工作区（默认 App Viewer）。
  - AC2：顶栏可见项目名称；可返回首页或项目列表。
  - AC3：输入区固定在对话区底部，支持发送。
  - AC4（P1）：顶栏可切换「应用查看器 / 编辑器 / 文件」等视图；未实现的视图需有明确空态，不得静默空白报错。

### 7.12 Plan 模式：Pat 澄清需求后执行

- 需求说明：Plan 开启时，Pat（产品）根据用户初始输入以提问方式收集需求，给出推荐路径（如 1 / 2 / 3 / 4）或允许用户自行定义，并输出需求规格说明供审阅；用户以 OK/确认等肯定性回复确认后，关闭 Plan 并静默开始执行——仅 Plan 时交工程师（Alex）；与 Team 同时开启时交 Leader（Mike）再任务分配；其后改码与预览闭环同现有路径。详见 `docs/superpowers/specs/2026-07-19-plan-mode-design.md`。
- 验收标准：
  - AC1：打开 Plan 并提交初始需求后，首轮由 Pat 回复（展示 **Pat | 产品**），而不是立刻由工程师改码。
  - AC2：Pat 通过提问收集关键需求信息（至少一轮有意义的澄清问答）。
  - AC3：Pat 给出可选择的推荐路径（编号选项），用户可通过输入编号（如 `1` / `2` / `3` / `4`）选择；也支持用户用自然语言自定义路径；路径对齐后给出需求规格说明并提示肯定性确认。
  - AC4：用户确认后，关闭 Plan；仅 Plan 时将确认需求交工程师执行，Plan∧Team 时交 Leader 任务流；此后进入改码 → 自动构建 → App Viewer 更新闭环（确认到下游为静默衔接，不强制再发一条手写执行指令）。
  - AC5：在需求未确认前，不应出现工程师已完成改码并触发成功构建的主路径（允许 Pat 纯对话澄清）。
  - AC6：同一初始需求在 Engineer 与 Plan 下的交互路径肉眼可区分。

### 7.13 记忆系统（细则待定）

- 需求说明：为 Agent 提供长期记忆能力，至少区分产品级与用户级；具体记忆类型、写入 / 检索 / 注入策略稍后讨论并回写本节 AC。
- 验收标准（阶段性占位，讨论后替换）：
  - AC1：存在可识别的记忆能力入口（领域包 / 端口或等价模块），且按登录用户隔离。
  - AC2：至少能持久化并在后续回合中被 Agent 使用一类记忆（具体类型待定）；刷新后仍可恢复。
  - AC3：详细类型划分（如 Product / Project / Decision / Knowledge / User Preference 等）与完整 AC 在设计讨论定稿后补充；未定稿前本条不作为阻断其它 P0 演示的硬门槛，但不得从 P0 列表删除。

### 7.14 Agent 安全与权限治理（P1）

- 需求说明：对 Agent 可调用的工具、可读写路径、可执行命令等施加显式权限边界；越权请求被拒绝且对用户 / 日志可观测。与 P2「开放注册与权限体系」（账号 / 多人协作 RBAC）区分：本条聚焦 **Agent 运行时能力治理**。
- 验收标准：
  - AC1：Agent 仅能调用已注册且当前模式 / 角色允许的工具；未授权工具不可用。
  - AC2：文件读写限定在当前项目 workspace（或明确允许的路径集合）内；越界路径被拒绝。
  - AC3：危险或未允许的命令 / 操作被拒绝，并产生可读失败原因（聊天提示或系统日志至少其一）。
  - AC4：权限策略可配置（如 YAML / 配置注册），不靠散落硬编码 if/else 堆砌。

### 7.15 Prompt 支持指定模型（P1）

- 需求说明：版本化 Prompt（`prompts/**` 的 meta 或等价配置）可声明推荐 / 指定模型；Agent 回合调用 LLM 时按该声明路由，而不是全局写死单一模型。
- 验收标准：
  - AC1：至少一份 Prompt 的 meta（或关联配置）可声明 `model`（或等价字段）。
  - AC2：使用该 Prompt 的回合实际请求的模型与声明一致（可用日志 / 测试断言验证）。
  - AC3：未声明模型时回退到全局默认模型，行为明确、可预期。
  - AC4：禁止在业务 TS 中为「换模型」硬编码大段 Prompt；模型绑定仍走 Prompt / 配置外置约定。

## 8. 成功标准

- 标准 A（笔试硬性）：公网可访问；真实交互；数据持久化；智能体驱动生成 + 可视化展示；非纯静态 PoC。
- 标准 B（范围完成度）：第 7 节全部 P0 验收标准通过（记忆系统以 7.13 阶段性 AC 为准，细则定稿后升级）。
- 标准 C（模式可感知）：评审人 2 分钟内能区分 Engineer、Team、Plan（及 Plan∧Team）的差异。
- 标准 D（闭环完整）：任意一次成功修改，均可观察到「改码 → 自动构建 → 预览更新」。
- 标准 E（公网安全底线）：无开放注册；未登录不可用核心能力；演示账号可文档说明。
- 标准 F（演示稳定性）：按「登录 → 新建 → 对话生成 →（Plan）澄清确认 /（Team）任务分配 → 预览更新 → 刷新仍在」路径可重复演示成功；回合进行中刷新不因断线误标失败。
